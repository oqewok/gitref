1/13 Вступление

Инструкции

Каждый шаг Обучения начинается с чтения текста. В конце текста располагается кнопка, выполняющая git команду.

После нажатия кнопки происходят 4 вещи в слудующем порядке:

  • В консоли (слева снизу): Команда будет добавлена и исполнена.
  • Справа: Анимация покажет, что сделает команда.
  • Когда анимация закончится, в консоль будет показан вывод (результат).
  • Справа от кнопки появится приглашение на следующий шаг обучения.

Официальное определение git – "распределенная система контроля версий". Что же это значит? Давайте посмотрим на примере.

В следующей таблице мы наблюдаем изменения файлов:

Время Статус
7 Июля, 10:00 Создан файл tasks.txt.
7 Июля, 10:05 Добавлен абзац tasks.txt.
J7 Июля, 10:10 Форматирование содержимого файла tasks.txt.
… …

Можно представить, что каждая строка таблицы является версией файла. Получается, что git это система, которая помогает нам следить за версиями или изменениями файлов сквозь время (что означает "распределенная" будет раскрыто далее по ходу обучения).

Основы git

Чтобы инициализовать git внутри директории, мы должны использовать команду git init. После исполнения данной команды, git создаст собственную директорию, храняющую данные для отслеживания изменений файлов.

У git есть три внутренних состояния (которые также называются областями): рабочая директория, индекс (иначе: область подготовленных файлов) и хранилище.

Нажмите на команду, а затем Далее.

git init Далее

2/13 Первые изменения

Чтобы увидеть git в действии, создадим обычный файл. Для этого можно исользовать команду touch и передать его имя в качестве первого элемента (Это команда ОС, а не git).

Эта команда создаст новый файл с именем tasks.txt. В этом файле будем хранить список заданий, которые необходимо сделать.

Если файл находится в области рабочей директории, то git следит за всем изменениями, которые произошли с файлом. Давайте скажем git моместить этот файл в область.

touch tasks.txt Далее

3/13 Добавление изменений в индекс

Коммит это изменение или группа изменений, которые мы объединяем в одно целое для хранения в системе контроля версий. Коммит создается в два шага: на превом в индекс добавляются все, которые мы хотим увидеть в индексе.

Для добавления изменений в индекс используется комада git add. Следующая команда перенесет изменения файла tasks.txt из рабочей директории в индекс.

git add tasks.txt Далее

4/13 Создаем свой первый коммит

Ранее мы видели первый шаг из двух на пути к созданию коммита.

Второй шаг явно указывает git совместить все изменения, в данный момент находящиеся в индексе (области подготовленных файлов), в единый узел коммита. Это делает команда git commit.

Каждому коммиту нужно давать емкий комментарий, описывающий произведенные изменения. Так мы сможем с легкостью просмотривать историю всех изменений в будущем. Чтобы оставить такое сообщение, после команды git commit используется ключ –m.

Следующая команда создаст наш первый коммит с комментарием указанного содержания. Добавлен файл tasks.txt.

git commit –m "Добавлен файл tasks.txt" Далее

5/13 Изменение файла

Команда echo вместе с >> Добавляет линии в конец файла.

Первая задача, которую мы хотим выполнить – Купить молока. Итак, добавляем ее в tasks.txt с помощью команды echo.

Look how git tells us that the file has changed by adding it to the working directory area. Посмотрим, как git горовит нам, что файл изменился, добавлением его в рабочую директорию.

echo "Купить молока" >> tasks.txt Далее

6/13 Git status

Другая полезная команда git status. Эта команда показывает нам все изменения в рабочем каталоге и в области слежения.

Посмотрим на вывод команды в консоли (в левом нижнем углу). git status показывает, что Изменения не отслеживаются индексом (это означает, что изменения есть, но никуда не занесены и остаются только внутри рабочей области). Все измененнные файлы внутри рабочей директори будут показаны в красном цвете. И файл tasks.txt имеет статус изменен, т.к. мы добавили к нему строку на предыдущем шаге.

git status Далее

7/13 Развернутый список изменений

Команда git status лишь показывает нам, какие файлы подверглись изменению. Если же мы хотим узнать, что изменилось в каждом конкретном файле из рабочей директории, мы можем использовать команду git diff.

В выводе команды git diff строки, что были добавлены в файл показаны зеленым цветом, а те, что были удалены, – красным.

git diff Далее

8/13 Добавление еще раз

Мы хотим создать другой коммит, вклчающий изменения в файле tasks.txt. Нужно добавить изменения в индекс вновь с помощью команды git add.

git add tasks.txt Далее

9/13 Git status 2

Используем команду git status еще раз, чтобы увидеть сооюбщение от git о том, что мы имеем изменения, занесенные в индекс. Файл tasks.txt все еще представлен со статусом "изменен", но цвет сообщения поменялся на зелёный, который означает, что изменения были внесены в индекс.

git status Далее

10/13 Развернутый список изменений 2

На шаге 7 мы использовали команду git diff для отображения детального списка изменений, имевшихся в рабочей директории.

Эта команда также может показать изменения и в индексе, если для нее передать ключ ––cached.

В выводе присутствуют сообщения тех же цветов: добаленные строки показаны зеленым, удаленные – красным.

git diff ––cached Далее

11/13 Наш второй коммит

Нашим новым коммитом станут все изменения, внесенные в индекс. Используем команду git commit вновь и передадим сообщение, описывающее этот коммит.

git commit –m "Добавлено первое задание" Далее

12/13 Просмотр коммитов

Команда, изучением которой заканчивается данное Обучение – git log. Эта команда показывает все сделанные коммиты. Давайте рассмотрим ее поближе.

git log имеет множество опций (ключей), с применением которых она может быть вызвана, что влияет на вывод и его детализацию. Давайте используем ключ ––oneline, чтобы увидеть вывод в минимальной детализации.

git log ––oneline Далее

13/13 Завершение

В данном Обучении была представлена система контроля версий git, раскрыта ее концепция использования: мы прошли через описание, сравнение ее с каталогом изменений. Было представлено большинство команд базовых команды, которые необходимо знать пользователю распределенной системы контроля версий для совершения первых удачных шагов в применении git в своих персональных проектах.
